# 需求文档

## 介绍

本功能旨在将 localsandbox 项目中的 Python sandbox SDK 从基于 microsandbox 服务器的实现改为基于 Docker 容器的本地实现。这个变更将使系统能够直接使用本地 Docker 容器来执行代码和命令，而不依赖于外部的 microsandbox 服务器，同时保持现有的 API 接口不变。

## 需求

### 需求 1

**用户故事：** 作为开发者，我希望 sandbox SDK 能够使用本地 Docker 容器而不是远程 microsandbox 服务器，以便减少外部依赖并提高执行性能。

#### 验收标准

1. 当初始化 sandbox 实例时，系统应该使用 Docker 命令行接口而不是 HTTP 请求来管理容器
2. 当创建 sandbox 时，系统应该能够启动相应的 Docker 容器（Python、Node.js 等）
3. 当停止 sandbox 时，系统应该能够正确清理和删除 Docker 容器
4. 如果 Docker 不可用，系统应该抛出清晰的错误信息

### 需求 2

**用户故事：** 作为系统管理员，我希望能够配置使用 Docker 或 Podman 作为容器运行时，以便支持不同的容器环境。

#### 验收标准

1. 当配置容器运行时时，系统应该支持通过环境变量或配置文件指定使用 docker 或 podman
2. 当未指定容器运行时时，系统应该默认使用 docker
3. 当指定的容器运行时不可用时，系统应该抛出配置错误
4. 如果容器运行时命令执行失败，系统应该提供详细的错误信息

### 需求 3

**用户故事：** 作为开发者，我希望现有的 API 接口保持不变，以便无需修改使用 sandbox SDK 的上层代码。

#### 验收标准

1. 当调用 `PythonSandbox.create()` 时，系统应该返回与原实现相同的接口
2. 当调用 `sandbox.run(code)` 时，系统应该返回相同格式的 `Execution` 对象
3. 当调用 `sandbox.command.run()` 时，系统应该返回相同格式的 `CommandExecution` 对象
4. 当调用 `sandbox.metrics` 相关方法时，系统应该返回相同格式的指标数据

### 需求 4

**用户故事：** 作为开发者，我希望能够执行 Python 代码并获得输出结果，就像原来的实现一样。

#### 验收标准

1. 当执行 Python 代码时，系统应该在 Docker 容器中运行代码并捕获标准输出
2. 当代码执行出错时，系统应该捕获错误输出并在 `Execution` 对象中正确标记
3. 当代码执行超时时，系统应该终止容器执行并返回超时错误
4. 如果容器启动失败，系统应该抛出 `RuntimeError` 异常

### 需求 5

**用户故事：** 作为开发者，我希望能够在 sandbox 中执行 shell 命令并获得结果，包括退出码和输出。

#### 验收标准

1. 当执行 shell 命令时，系统应该在 Docker 容器中运行命令并捕获输出
2. 当命令执行完成时，系统应该正确返回退出码、标准输出和错误输出
3. 当命令执行超时时，系统应该终止执行并返回超时信息
4. 如果命令不存在，系统应该返回相应的退出码和错误信息

### 需求 6

**用户故事：** 作为开发者，我希望能够获取 sandbox 的资源使用指标，以便监控系统性能。

#### 验收标准

1. 当调用 `sandbox.metrics` 时，系统应该返回一个 TODO 标记，表示此功能暂未实现
2. 现有的 metrics 实现应该被移除或标记为待实现
3. BaseSandbox 中的 metrics 属性应该标记为 TODO
4. 系统应该能够正常运行而不依赖 metrics 功能

### 需求 7

**用户故事：** 作为开发者，我希望支持不同类型的 sandbox（Python、Node.js），每种类型使用相应的 Docker 镜像。

#### 验收标准

1. 当创建 PythonSandbox 时，系统应该使用 Python 相关的 Docker 镜像
2. 当创建 NodeSandbox 时，系统应该使用 Node.js 相关的 Docker 镜像
3. 当指定自定义镜像时，系统应该使用指定的镜像而不是默认镜像
4. 如果指定的镜像不存在，系统应该尝试拉取镜像或返回错误

### 需求 8

**用户故事：** 作为开发者，我希望能够配置容器的资源限制（内存、CPU），以便控制资源使用。

#### 验收标准

1. 当指定内存限制时，系统应该在创建容器时设置相应的内存限制
2. 当指定 CPU 限制时，系统应该在创建容器时设置相应的 CPU 限制
3. 当指定卷映射时，系统应该正确挂载主机目录到容器中
4. 如果资源限制设置失败，系统应该返回清晰的错误信息

### 需求 9

**用户故事：** 作为开发者，我希望系统能够正确处理容器生命周期，包括创建、运行和清理。

#### 验收标准

1. 当 sandbox 启动时，系统应该创建并启动相应的 Docker 容器
2. 当 sandbox 停止时，系统应该停止并删除 Docker 容器
3. 当系统异常退出时，系统应该能够清理遗留的容器
4. 如果容器已经存在同名容器，系统应该先清理旧容器再创建新容器

### 需求 10

**用户故事：** 作为开发者，我希望保持异步编程模型，以便与现有的异步代码兼容。

#### 验收标准

1. 当调用任何 sandbox 方法时，系统应该使用 async/await 模式
2. 当执行长时间运行的操作时，系统应该不阻塞事件循环
3. 当多个操作并发执行时，系统应该正确处理并发访问
4. 如果异步操作被取消，系统应该正确清理资源
