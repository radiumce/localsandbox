# Core Definition
The sandbox is an allocated remote/isolated environment used for executing code and installing software. You can only access and receive feedback from it via **CLI tool invocations** and **file reading/writing**. There is no GUI access.

# Tool Stack
- **`lsb-cli` Client**: Core independent command-line interface used to manage sandbox environments, session lifecycles, code execution, and file mappings.
- **Checklist Tool**: Should be used to plan and track execution when tasks are complex.

# Part 1: Session Management
To ensure context consistency across executions, explicitly follow these session rules:

1. **Context Continuity**
   - You must reuse the same `session_id` when performing a sequence of related tasks.
   - Do not frequently switch or create new sessions without reason, as this drops environmental variables and temporary files.

2. **Timeout & Expiration Handling**
   - Sandbox Sessions have an inactivity timeout.
   - **Action**: If you encounter an expiration or timeout error, immediately generate a new session (by executing a command without a session ID) and explicitly re-configure any necessary dependencies.

# Part 2: Execution Standards

1. **Pre-flight Check (Permissions)**
   - Verify the current user identity before complex tasks.
   - Some sandboxes might not run as root. If system configuration or global software installation is required, check if `sudo` is necessary.

2. **Output Hygiene**
   - **Success**: Strictly control `stdout` length. It is recommended to output only the end of long logs (e.g., `tail -n 20`) to avoid consuming excessive Context Window.
   - **Failure**: If a command fails, you MUST fetch and analyze the complete `stderr` output for debugging.

3. **Implicit Execution**
   - **Action**: When tasked with programming, directly execute the code in the sandbox.
   - **Constraint**: **DO NOT** output the raw source code you wrote into the chat interface unless the user explicitly asks to see it. Just run it and report the outcome.

# Part 3: CLI Usage (`lsb-cli`)

The `lsb-cli` tool provides a comprehensive interface for managing sandboxes. Below are the primary commands and usage examples:

## 1. Executing Code (`exec-code`)
Executes a raw code snippet (default: Python).
- **Basic Execution**: `lsb-cli exec-code -c "print('Hello World')"`
- **Reuse Session**: `lsb-cli exec-code -c "print(x)" -s <session_id>`
- **Read from Stdin**: `cat script.py | lsb-cli exec-code -c -`
- **Other Languages**: `lsb-cli exec-code -c "console.log('JS')" -t node`

## 2. Executing Shell Commands (`exec-cmd`)
Executes standard shell commands inside the sandbox.
- **Basic Command**: `lsb-cli exec-cmd -c "ls -la /shared"`
- **Chained Commands**: `lsb-cli exec-cmd -c "pip install requests && python main.py" -s <session_id>`

## 3. Persistent Sandbox (`pin` & `attach`)
Allows you to save a sandbox from garbage collection by giving it a human-readable name, and attach to it later.
- **Pin a Sandbox**: `lsb-cli pin <session_id> my-project-env`
- **Attach to Pinned**: `lsb-cli attach my-project-env` (Returns the underlying Session ID)

## 4. Session & Resource Management
- **List Active Sessions**: `lsb-cli sessions`
- **List Volume Mappings**: `lsb-cli volumes`
- **Force Stop Session**: `lsb-cli stop <session_id>`

# Part 4: File System & Mapping

Sandbox internal files are isolated from the Host file system. You must use mapped volumes to interact.

1. **Retrieve Mappings**
   - Call `lsb-cli volumes` to get the mapping between "Host Path" and "Sandbox Path".
   - **Example**: `Host: /Users/Me/project -> Container: /shared`

2. **File Interaction Workflow**
   - If you need to read a file generated by the sandbox (e.g., an image, a compiled binary):
     1. Write the file to the mapped directory inside the sandbox (e.g., `/shared/output.png`).
     2. Deduce the Host path using the mapping (e.g., `/Users/Me/project/output.png`).
     3. Use the Host's file system tools to read or move the file.

# Troubleshooting
- **Input Validation**: If a program crashes, first verify the input arguments and environment dependencies.
- **Output Correction**: If results don't align with expectations, consider adjusting CLI arguments or library usage, and verify the changes inside the sandbox.
